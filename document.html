bot-mua-hang

An AI-powered Shopee shopping bot that reads orders from an Excel file and autonomously adds products to cart using a Vision LLM with element-level detection. All order data is persisted in SQLite with multi-session support.

Python 3.11+ Patchright (Playwright) Gemma-4-31B-IT (VNG Cloud) FastAPI + SQLite shopee.vn

Business context

VNG marketing team sends gifts to users. Previously, the purchasing team manually bought each gift on Shopee — taking up to 60 days. This bot automates the add-to-cart process by selecting the correct product variants (size, color) and adding items to the shopping cart, reducing manual effort significantly.

How it works

  1. Import ordersUpload an Excel file via the dashboard UI. Each row is one gift order with product URL, variant/size/color, and quantity.
  2. Login to ShopeeEither use CDP mode (connect to your real Chrome) or click "Login to Shopee" in the dashboard. Session is saved to disk.
  3. Click StartThe bot opens each product URL, extracts interactive elements via JavaScript, and the Vision LLM picks elements by number to select variants and add to cart.
  4. Status persisted to SQLiteAfter each order, the status, screenshots, and step-by-step reasoning are saved to data/bot.db. All data persists across restarts and is browsable in the dashboard.

The FastAPI server runs in a background daemon thread. The bot loop runs in the main asyncio event loop. They communicate through AppState — a shared singleton that uses threading.Event objects for cross-thread signalling and asyncio.Queue for WebSocket broadcasts.

CDP mode (recommended)

For best anti-bot evasion, connect to your real Chrome browser instead of launching a new one. Your real Chrome has natural fingerprint, browsing history, and Shopee trust score — making CAPTCHA far less likely.

  1. Start Chrome with debugging macOS: /Applications/Google\ Chrome.app/Contents/MacOS/Google\ Chrome --remote-debugging-port=9222
    Linux: google-chrome --remote-debugging-port=9222
    Windows: "C:\Program Files\Google\Chrome\Application\chrome.exe" --remote-debugging-port=9222
  2. Log in to ShopeeIn that Chrome window, go to shopee.vn and log in normally with your account.
  3. Set CDP endpointIn config.yaml, set cdp_endpoint: "http://localhost:9222"
  4. Run the botpython main.py --file orders.xlsx — the bot attaches to your existing Chrome. No new browser window opens.
Important — in CDP mode, the bot uses the first tab of your existing Chrome. The session_dir and headless settings are ignored. When the bot finishes, it disconnects without closing your Chrome.

Shopee login & session persistence

The bot supports two login modes depending on your configuration:

CDP mode (recommended)

Log in to Shopee directly in your Chrome browser before starting the bot. The bot inherits your existing session — no login flow needed.

Launch mode (non-CDP)

The bot uses a persistent browser profile stored in ./session/. Once you log in, the session is saved and reused on every subsequent run.

  1. Start the botpython main.py — browser launches immediately.
  2. Check dashboard headerThe login badge shows ✗ Not logged in.
  3. Click "Login to Shopee"The browser navigates to shopee.vn/buyer/login.
  4. Log in normallyEnter your phone/email and password. Complete any Shopee captcha manually.
  5. Dashboard updatesWithin 3 seconds the badge turns ✓ Logged in. Session is now saved to ./session/.
Reset session — to log in as a different account, delete the session/ directory and re-run.

Add to Cart flow

After navigating to the product URL, the bot uses a hybrid JavaScript + Vision LLM pipeline to add the product to cart. A typical successful flow completes in 3–4 steps (~35 seconds):

  1. Select color/variantJavaScript extracts all interactive elements with pixel coordinates. The LLM identifies the correct color button by its text label (e.g. "Đen") and clicks it. JavaScript detects the orange border indicating selection.
  2. Select sizeThe LLM sees the color is now marked [SELECTED] and picks the correct size button (e.g. "M"). JavaScript confirms selection via border color detection.
  3. Add to CartWith all variants marked [SELECTED], the LLM clicks the "Thêm Vào Giỏ Hàng" (Add To Cart) button. The bot does NOT click "Mua Ngay" (Buy Now).
  4. Confirm successThe LLM sees the success toast/modal ("Sản phẩm đã được thêm vào Giỏ Hàng") and returns action="done".
Add to Cart only — the bot stops after adding to cart. It does NOT proceed to checkout, payment, or order placement. This is by design for the current workflow.

Automatic failure cases

The LLM immediately returns action="fail" for:

  • CAPTCHA / anti-bot screen detected
  • Login page (session expired)
  • Product out of stock or unavailable
  • Access denied

Stuck detection

If the bot repeats the same action 3 times in a row (same action type + same element), it triggers a recovery: press Escape twice, scroll to top, then re-analyze the page.

OTP handling

When Shopee requires a one-time password (e.g. during login or payment), the LLM returns action="wait_otp". The bot then:

  1. Pauses the loopThe order status changes to waiting_otp.
  2. OTP modal appearsA full-screen modal pops up in the dashboard with an OTP input field (auto-focused).
  3. User enters OTPType the code received by SMS/email, press Enter or click Submit. 5-minute timeout.
  4. Bot continuesThe OTP is typed into the browser and processing resumes automatically.

Re-buy

The Re-buy button on each order row lets you re-process any order that already has a success or failed status. It:

  • Resets the order status to pending in the Excel file
  • Reloads the order from Excel (so any edits you made are picked up)
  • Queues it for immediate re-processing

The bot stays alive indefinitely after finishing all orders, waiting for re-buy requests — no restart needed.

Module reference

src/agent.py — VisionAgent

Wraps the VNG Cloud OpenAI-compatible API. Uses a Set-of-Mark (SoM) approach: draws numbered red badges on each detected element in the screenshot, then sends the annotated image + element list to Gemma-4-31B-IT. The LLM picks elements by number (element_index) rather than guessing pixel coordinates.

Action typeMeaning
clickClick the element at element_index (coordinates from JavaScript)
typeClick element, then type text_to_type, press Enter
scrollScroll up or down by 540px (in small natural increments)
press_escapePress Escape key to close modals/popups/overlays
wait_otpPause and request OTP from user
doneItem successfully added to cart
failUnrecoverable error — mark order as failed

Key functions:

  • _draw_markers(screenshot, elements) — draws numbered badges on screenshot; orange for selected elements, red for normal
  • VisionAgent.decide(screenshot, objective, elements) — sends annotated screenshot + element list to LLM, returns Action
  • Action.from_dict(data, elements) — maps element_index to pixel coordinates from the elements list

src/browser.py — BrowserManager

Manages browser automation with two connection modes and human-like interaction primitives.

Connection modes

  • CDP mode (cdp_endpoint set) — connects to existing Chrome via connect_over_cdp(). Uses the first context and page. Sets viewport to configured size.
  • Launch mode (cdp_endpoint null) — launches patchright's own Chromium with persistent user data dir.

Human-like behavior

  • _human_move(x, y) — cubic Bézier curve from current position to target with 20–40 steps, random control points, and ±1px jitter per step
  • _human_click(x, y) — Bézier move, then mouse.down() / mouse.up() (not click()) for proper event sequence
  • _human_type(text) — character-by-character with 50–150ms delays, occasional 200–400ms thinking pauses
  • _human_scroll(delta) — small incremental mouse.wheel() calls with random step sizes (40–120px) and pauses

Element extraction

extract_elements() runs JavaScript in the page to find interactive elements. Returns a list of {t, x, y, w, h, s} dicts:

  • t — element text (max 60 chars)
  • x, y — center coordinates (pixels)
  • w, h — bounding box dimensions
  • s — selected state (boolean, detected via orange/red border color)

Two-pass scan: first button, [role="button"], input, textarea; if <30 found, also scans div, span, a, label with cursor: pointer.

src/shopee_bot.py — ShopeeBot

Orchestrates one order at a time: navigate → screenshot + element extraction → LLM decision → execute action → loop. Includes stuck detection (3 identical actions → Escape + scroll to top), CAPTCHA URL detection, and comprehensive debug logging with per-step screenshots.

src/excel_handler.py

Three functions: load_orders(path) returns all rows as dicts with _row_index. update_order_status(path, row, status, order_id, note) writes the result columns. update_order_fields(path, row, fields) writes arbitrary column updates (used by the Edit endpoint).

src/state.py — AppState

Global singleton shared between the bot loop and the FastAPI server. Contains the orders list, all event primitives, and broadcast() which pushes the serialized state to all connected WebSocket clients.

Vision pipeline

The bot uses a hybrid JavaScript + LLM approach that separates what to do (LLM) from where things are (JavaScript):

Screenshot (raw PNG)
    │
    ├── JavaScript: extract_elements()
    │     ├── Finds buttons, inputs, clickable elements
    │     ├── Gets bounding box centers (exact pixels)
    │     └── Detects selection state via border color
    │            isWarm(rgb) → R>200, G<150, B<100
    │
    ├── Python: _draw_markers()
    │     ├── Draws numbered badges on screenshot
    │     ├── Red badge = normal element
    │     ├── Orange badge = [SELECTED] element
    │     └── Green/orange outline around element bounds
    │
    └── LLM: Gemma-4-31B-IT
          ├── Sees annotated screenshot + element text list
          ├── Picks element_index by NUMBER (not coordinates)
          ├── Trusts [SELECTED] markers (no visual detection needed)
          └── Returns JSON: {action, element_index, reasoning}

Why this approach?

Earlier iterations tried having the LLM estimate pixel coordinates or grid cells directly from screenshots. Gemma-4-31B-IT's spatial reasoning was consistently off by 200+ pixels — it would click the wrong variant button or miss the Add to Cart button entirely. By using JavaScript for precise element location and letting the LLM only decide which element (by number), accuracy went from ~30% to nearly 100%.

Selection state detection

Shopee indicates selected variants (color, size) with an orange/red border. The LLM could not reliably detect this visual cue in screenshots. Instead, JavaScript checks the computed borderColor and outlineColor CSS properties using isWarm(): if R>200, G<150, B<100, the element is marked [SELECTED] in the element list. The LLM trusts this marking and never re-clicks a selected element.

Anti-bot strategy

Standard Playwright is blocked by Shopee. Shopee's anti-bot system detects CDP (Chrome DevTools Protocol) fingerprints at the browser binary level — JS-level patches alone are not enough.
Detection vectorHow we handle it
CDP binary fingerprintPatchright patches the Chromium binary before compilation — markers are removed at source level
Browser fingerprint / trust scoreCDP mode: connect to user's real Chrome with natural browsing history, extensions, and cookies — much higher trust score than a fresh Chromium
navigator.webdriverRemoved by patchright (binary level)
Robotic mouse movementCubic Bézier curves with random control points, 20–40 intermediate steps, ±1px jitter
Instant clicksmouse.down() / mouse.up() with 50–120ms hold delay, preceded by natural mouse movement to the target
Robotic typingCharacter-by-character with 50–150ms delays, occasional 200–400ms "thinking" pauses
Instant scrollingSmall incremental mouse.wheel() calls (40–120px steps) with 15–50ms pauses
Predictable timingRandom jitter (±500ms) added to all wait times between actions
Sequential orders (rate limiting)Orders processed one at a time with 2.5–3.5s base delays between actions
Locale / languageBrowser launched with locale="vi-VN"

Session persistence

Launch mode (non-CDP)

Patchright's launch_persistent_context(user_data_dir="./session") saves the full Chromium profile — cookies, localStorage, IndexedDB — to the session/ directory. This means:

  • Shopee login persists across bot restarts
  • Saved credit cards and addresses are available
  • To switch accounts: rm -rf session/ and re-login
  • The session/ directory is gitignored

CDP mode

Session is managed by your real Chrome browser. The bot simply connects and disconnects — it does not modify your Chrome profile or close Chrome when it exits.
